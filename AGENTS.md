# Agent Instructions

## Project Overview

ISTA-Patcher is a GPL-3.0-or-later IL patcher for BMW ISTA diagnostic software.
It patches .NET assemblies using `dnlib` and exposes both a CLI and an Avalonia GUI.

## Repository Layout

```text
src/ISTA-Patcher/   CLI application: DotMake.CommandLine, Serilog, Sentry, GenHTTP
src/ISTAvalon/      GUI application: Avalonia 12, CommunityToolkit.Mvvm
src/ISTAlter/       Core patching library: dnlib, SkiaSharp, platform utilities
src/ISTgenerAtor/   Roslyn source generator used as an analyzer by ISTAlter
src/ISTestA/        NUnit 4 tests, including Avalonia headless GUI tests
openspec/           Structured change proposals, specs, and archived changes
scripts/            Release, coverage, and repository validation helpers
```

Important relationships:

- `ISTA-Patcher` and `ISTAvalon` both depend on `ISTAlter`.
- `ISTAlter` references `ISTgenerAtor` as an analyzer-only project reference.
- `ISTestA` references `ISTAlter` and `ISTAvalon`.
- `packages.lock.json` is committed per restorable project.

## Architecture Notes

- `ISTAlter.Core.Patch` orchestrates patching. `PatchUtils` is a `static partial` class; some members are generated by `ISTgenerAtor`.
- CLI commands live in `src/ISTA-Patcher/Commands/` and use DotMake.CommandLine attributes with kebab-case, double-hyphen options.
- `Global.Config` loads `appsettings.json`; `Global.LevelSwitch` is shared by CLI and GUI logging.
- `ISTAvalon.Services.CommandDiscoveryService` reflects CLI command descriptors and builds GUI tabs dynamically.
- GUI ViewModels derive from `CommunityToolkit.Mvvm.ComponentModel.ObservableObject`.
- GUI logs flow through `DelegateLogSink` into each tab's right-side log panel.
- `GuiStateDumper` serializes the GUI state as YAML. When enabled, `GuiObservationServer` exposes it on localhost through `GET /dump.yaml`.

## Build And Test

Use the .NET SDK configured by `global.json` (.NET 10, prerelease allowed).

```bash
dotnet restore
dotnet build
dotnet build -c Release
dotnet test
dotnet test src/ISTestA/ISTestA.csproj --filter FullyQualifiedName~UIAutomationTests
bash scripts/check_copyright_years.sh
```

Run the CLI:

```bash
dotnet run --project src/ISTA-Patcher -- patch [options] "/PATH/TO/ISTA"
```

Run the GUI:

```bash
dotnet run --project src/ISTAvalon
```

Run the GUI with local dump observation enabled:

```bash
dotnet run --project src/ISTAvalon -- --gui-dump-http
curl http://127.0.0.1:8765/dump.yaml
```

Observation options:

```text
--gui-dump-http
--gui-dump-http-port <port>
--gui-dump-http-host <host>
ISTA_GUI_DUMP_HTTP=1
ISTA_GUI_DUMP_HTTP_PORT=8765
ISTA_GUI_DUMP_HTTP_HOST=127.0.0.1
```

Keep the observation endpoint local-only unless there is a clear reason to do otherwise.

## Code Style

Every `.cs` file must begin with SPDX headers:

```csharp
// SPDX-License-Identifier: GPL-3.0-or-later
// SPDX-FileCopyrightText: Copyright YYYY[-YYYY] TautCony
```

Project conventions:

- Use file-scoped namespaces.
- Put `using` directives after the file-scoped namespace declaration.
- Keep nullable annotations enabled according to each project file.
- Prefer existing services, models, and ViewModel patterns over new abstractions.
- Do not hand-edit generated members that belong to `ISTgenerAtor`.
- Preserve locked package files when package references change.
- Avoid unrelated formatting churn.

Analyzer setup:

- `StyleCop.Analyzers` and `Roslynator.Analyzers` are used across production projects.
- `ISTgenerAtor` also uses Roslyn analyzer packages and targets `netstandard2.0`.
- `ISTAlter` allows unsafe blocks for low-level patching utilities.

## GUI Guidelines

- `MainWindow.axaml` defines parameter editor templates for `Bool`, `Enum`, `Numeric`, `Path`, `String`, and `StringArray` ViewModels.
- Adding a CLI command or option should usually surface in the GUI through reflection; update GUI-specific code only when the command metadata is insufficient.
- Keep GUI state observable through `GuiStateDumper` when adding visible UI, especially toolbar controls, tabs, log panel behavior, validation state, and bound parameter values.
- Headless tests can print YAML dumps and save screenshots under the NUnit work directory.

## Tests

- Core tests live under `src/ISTestA/ISTAlter/`.
- GUI/ViewModel/headless tests live under `src/ISTestA/ISTAvalon/`.
- Prefer focused tests for behavior changes. Broaden test coverage when touching command discovery, command execution, GUI binding, or shared patching logic.
- `dotnet test --no-restore` can fail after project or package changes if generated Avalonia resources are stale; run a normal `dotnet test` before treating that as a product failure.

## OpenSpec Workflow

Structured change management lives under `openspec/`:

```text
openspec/specs/             Stable living specs
openspec/changes/           Active changes
openspec/changes/archive/   Completed changes
```

Use OpenSpec artifacts when the change is feature-sized or affects documented behavior:

- `proposal.md` describes intent and scope.
- `design.md` captures architecture and tradeoffs when needed.
- `tasks.md` tracks implementation.
- `specs/*/spec.md` contains requirement deltas.

Keep implementation, tests, and specs aligned before archiving a change.

## Safety Notes

- This project patches third-party installation files. Preserve backup and path-safety behavior.
- Be careful with traversal, overwrite, and platform-specific filesystem logic.
- Do not expose local GUI observation or GenHTTP server endpoints beyond localhost without explicit intent.
- Sentry is configured through `appsettings.json`; avoid logging secrets, local license data, or full sensitive paths unless already part of expected diagnostics.
